## 1. 本ドキュメントの目的

プロアクティブ提案エンジンは iAgent の核心機能である。
「何を」「いつ」「どう」提案すべきかの解像度を上げるため、3 人のユーザーペルソナを設定し、具体的なユースケースから機能要件を導出する。

## 2. ペルソナ定義

### 2.1 ペルソナ A: 高橋 遼太（32歳）— テックリードエンジニア / 情報ヘビーコンシューマー

| 属性 | 内容 |
|------|------|
| **職業** | SaaS スタートアップのテックリード（チーム 8 名） |
| **デバイス** | MacBook Pro（業務）、iPhone（移動中）、自宅 PC（個人開発） |
| **使用サービス** | GitHub, Slack, Notion, Zenn, Hacker News, arxiv, lobste.rs, Reddit r/programming |
| **AI 馴染み度** | 高い。GitHub Copilot / ChatGPT を日常的に使用 |
| **情報行動** | RSS 80+ フィード購読（技術ブログ、OSS リリース、セキュリティアドバイザリ、業界ニュース等）。**1 日あたり平均 200 件の新着記事が流入**。Hacker News / lobste.rs を 1 日 5 回以上チェック。週末に arxiv の ML/Systems 論文を巡回 |
| **意思決定スタイル** | データドリブン。根拠を集めてから判断する。情報が足りないと意思決定を保留する傾向 |

**ペインポイント:**
- **1 日 200 件の RSS が処理しきれない**。全部目を通す時間はなく、タイトルだけで判断すると重要記事を見逃す
- 情報ソースが多岐にわたり（RSS + HN + lobste.rs + X + arxiv）、**同じトピックが複数ソースに分散**。統合的に把握できない
- 競合サービスの動向（プライシング変更、新機能リリース等）の把握が後手に回る
- 「今週の自分の関心事に関連する記事」と「汎用的に重要な記事」の区別がつかない
- チームメンバーに共有すべき情報と自分だけ知っていればいい情報の仕分けが手間
- 技術選定時に「前に読んだあの記事」が見つからない。1 日 200 件のうち記憶に残るのは 2-3 件
- 情報過多による **意思決定疲れ** — 選択肢が多すぎて判断が遅れる

**iAgent への期待:**
- 200 件/日の RSS を **多段階でフィルタリング** し、「必読」「目を通すべき」「スキップ可」に仕分けてほしい
- 複数ソースに分散した同一トピックの情報を **統合・重複排除** してほしい
- 今週の関心事（技術選定、調査テーマ等）に合わせて **フィルタの重みを動的に調整** してほしい
- 競合の Web ページの変更を検知したら即通知してほしい
- 過去にクリップした記事や保存した知見を、関連する会話の文脈で自動的に引き出してほしい
- 「今日の 200 件を全部読んだ人」のように振る舞い、聞けば答えてくれる **情報番人** になってほしい

### 2.2 ペルソナ B: 山田 美咲（38歳）— プロジェクトマネージャー

| 属性 | 内容 |
|------|------|
| **職業** | IT コンサルティング企業の PM（同時 3 プロジェクト掛け持ち） |
| **デバイス** | Windows ノート PC（業務）、iPhone（常時携帯） |
| **使用サービス** | Google Calendar, Teams, Backlog, Confluence |
| **AI 馴染み度** | 中程度。ChatGPT は使うが、自動化ツールの設定は苦手 |
| **情報行動** | メール/チャットの通知に追われる日々。情報は人経由で入ってくることが多い |
| **意思決定スタイル** | 直感 + 経験ベース。素早く判断し、修正しながら進める |

**ペインポイント:**
- 3 プロジェクトの MTG が 1 日 5-8 件。次の会議の準備が間に合わない
- 「来週のクライアント報告に必要なデータ」を思い出すのが直前になる
- プロジェクト横断のタスクの優先順位を見失いがち
- 帰宅後に「あ、あれ忘れてた」が頻発

**iAgent への期待:**
- 朝一で今日の全体像（予定 + 準備すべきこと）を教えてほしい
- 会議 30 分前に「この会議で必要な情報」をリマインドしてほしい
- 自分が設定した目標（期日付きタスク）の進捗を定期的に確認してほしい
- エージェントに話した「やること」を忘れずに拾ってほしい

### 2.3 ペルソナ C: 佐藤 陽菜（25歳）— 社会人 3 年目 / 個人学習者

| 属性 | 内容 |
|------|------|
| **職業** | Web 制作会社のデザイナー。フロントエンド技術の習得を目指している |
| **デバイス** | MacBook Air（業務+個人）、iPhone（主利用） |
| **使用サービス** | Figma, YouTube, Udemy, Zenn, X（旧 Twitter） |
| **AI 馴染み度** | 低〜中。ChatGPT で質問はするが、活用の幅は狭い |
| **情報行動** | 学習コンテンツを「あとで見る」に入れて忘れる。X で流れてきた記事を衝動的に読む |
| **意思決定スタイル** | 興味駆動。ハマると深掘りするが、飽きると別のことに移る |

**ペインポイント:**
- 学習計画を立てても 2 週間で形骸化する
- 「先月学んだこと」を振り返る習慣がなく、定着しない
- 興味が分散し、何から手をつければいいか分からなくなる
- 成長実感が得られず、モチベーションが波打つ

**iAgent への期待:**
- 学習の進捗を可視化し、継続を後押ししてほしい
- 「今日は React の続きをやろう」のような日次のナッジがほしい
- 過去に学んだことと今の関心をつなげて「次に学ぶべきこと」を提案してほしい
- 週次/月次で振り返りをまとめ、成長を実感させてほしい

## 3. ユースケースシナリオ

### 3.1 ペルソナ A: 高橋 遼太 — 情報ヘビーコンシューマーの 1 日

```
07:00  起床。iPhone で iAgent の朝のブリーフィングを確認
       → [提案 A1] 「昨夜〜今朝の RSS 新着 203 件を分析しました:
         ■ 必読（5 件）: ORM 選定に直結する記事 + セキュリティアドバイザリ
         ■ 推奨（12 件）: 関心テーマに関連。時間があれば
         ■ トピックまとめ: 'Bun 1.2 リリース' が 4 ソースで言及（統合済み）
         ■ スキップ: 186 件（根拠付き）」
       → [提案 A2] 「監視中の競合サービスページに変更あり（プラン改定）」

09:00  出社。「今朝の必読 5 件の詳細教えて」
       → [提案 A3] 各記事の 3 行要約 + 自分に関係する理由 + 元記事リンク
       → [提案 A4] 「うち 2 件は ORM 選定に関連。先週のクリップ 4 件と合わせて
         Drizzle の情報が充実してきています」

10:00  アーキテクチャレビュー会議中
       → 「Prisma と Drizzle の比較、最近の情報まとめて」
       → [提案 A5] 過去 2 週間の RSS 38 件 + クリップ 6 件 + 会話メモから
         論点を自動整理して比較表を生成

12:00  昼休み。「今朝の推奨 12 件、ざっくり何があった？」
       → [提案 A6] 12 件をカテゴリ別に要約（インフラ 4 件、フロントエンド 3 件、
         セキュリティ 2 件、キャリア/コミュニティ 3 件）

15:00  技術調査中。「最近 Edge Runtime 関連で何か動きあった？」
       → [提案 A7] 直近 1 週間の RSS + クリップから Edge Runtime 関連を横断検索
         → 該当 8 件の時系列サマリー + 論点整理

18:00  退社前。明日の準備
       → [提案 A8] 「今日の情報まとめ: 必読 5 件中 3 件消化、推奨 12 件中 4 件閲覧。
         ORM 選定の情報蓄積が進んでいます（来週月曜に最終決定）。
         明日注目: Cloudflare の開発者カンファレンスがあり、RSS 増加が予想されます」

23:00  ふりかえり自動実行
       → reflection: 「ORM 選定が Drizzle 方向で収束。今日の 203 件のうち
         実際に価値があったのは必読 5 件 + 会議中の横断検索。
         フィルタリング精度: 必読の Accept 率 100%、推奨は 33%（12 件中 4 件閲覧）
         → 推奨の閾値を上げるべき」
       → 翌日のフィルタリング重みを自動調整
```

**提案が邪魔になる場面:**
- 集中コーディング中に RSS 新着の通知が来る → **フォーカスモード時は提案を抑制すべき**
- 既に読んだ記事を再度提案される → **既読/クリップ済みの重複排除が必要**
- 重要度の低い RSS 更新（typo修正レベル）を通知される → **変更の重要度判定が必要**
- 同じトピックが複数ソースから別々に提案される → **トピック統合・重複排除が必須**
- 200 件全部の一覧を見せられる → **未フィルタの大量情報は提案ではなくノイズ**

### 3.4 中長期シナリオ（1 週間〜1 ヶ月）

#### ペルソナ A — 1 ヶ月で技術選定を完了（累計 RSS 約 6,000 件を処理）

```
Week 1: 情報収集フェーズ（RSS 約 1,400 件 → 必読 32 件に精選）
  - 毎朝の 200 件から ORM 関連を自動抽出。1 週間で ORM 関連 18 件を蓄積
  - 複数ソースのクロスリファレンス: 「Drizzle v2.1」が 6 ソースで言及 → 統合サマリー

  - reflection: 「ORM 選定の情報収集が活発。Prisma/Drizzle/TypeORM が候補。
    RSS フィルタリング精度: 必読 Accept 率 87%、推奨は 30% → 推奨の閾値を調整」
  - **フィルタ学習**: Accept された記事の特徴（ベンチマーク、実運用報告）を次週の重み付けに反映

Week 2: 比較・評価フェーズ（RSS 約 1,400 件 → ORM 関連のみ深掘り）
  - [提案] 「2 週間で ORM 関連 38 件（RSS 18 + クリップ 6 + HN/lobste.rs 14）を蓄積。
    3 候補の比較表を作成しましょうか？」
  - ソース横断で論点を自動分類: パフォーマンス / 型安全性 / エコシステム / 移行コスト
  - ORM 以外のトピックもフィルタは継続 — 「セキュリティアドバイザリ」「チーム関心事」は並行追跡
  - reflection: 「推奨の Accept 率が 42% に改善（先週 30%）。ベンチマーク記事の重みを上げた効果」

Week 3: チームへの共有（フィルタ精度が安定期に）
  - [提案] 「比較表をベースに ADR ドラフトを作りますか？
    過去の reflection から 'パフォーマンスを重視する傾向' を検出。
    ベンチマーク情報を優先配置します」
  - [提案] 「今週の RSS 1,400 件から、チーム共有に値する記事 3 件をピックアップ:
    (1) Bun のセキュリティ脆弱性 (2) TypeScript 6.0 ロードマップ (3) Vercel の料金改定」

Week 4: 決定・振り返り
  - reflection: 「Drizzle を採用。決定要因はバンドルサイズと型安全性。
    4 週間の情報処理: RSS 約 5,600 件 → 必読 128 件 → ORM 関連 38 件 → 最終決定。
    フィルタリング精度推移: 必読 Accept 率 87% → 93%。推奨は 30% → 48%」
  - 知見を長期メモリに構造化保存
  - 選定プロセス全体を procedural memory 候補として記録
  - ORM 関連のフィルタ重みをベースラインに戻す（選定完了のため）
```

## 4. 機能要件の抽出

### 4.1 提案の類型

ユースケースから抽出した全提案を分類する。

| 類型 | 説明 | 該当提案 | 頻度 |
|------|------|---------|------|
| **ダイジェスト型** | 大量の情報を要約・精選して提示 | A1, B7, C8 | 日次/週次 |
| **アラート型** | 重要な変化を即時通知 | A2 | イベント駆動 |
| **リマインド型** | 予定・タスクの事前通知 | B1, B3, B4, B8 | イベント接近時 |
| **コンテキスト型** | 会話の文脈に合わせた関連情報の提示 | A4, A5, A7, C5, C6 | チャット時 |
| **ナッジ型** | 行動の後押し・継続促進 | C1, C4, C7 | 日次 |
| **整理提案型** | 蓄積情報の整理・構造化を提案 | A6, C3 | 閾値到達時 |
| **進捗追跡型** | 目標・計画の進捗を可視化 | B5, B6, C2 | 定期 |
| **フォローアップ型** | 前回の文脈を引き継いだ継続提案 | A3, A8 | 状態変化時 |

### 4.2 トリガー条件の整理

| トリガー | 説明 | 実現方法 |
|---------|------|---------|
| **時刻ベース** | 固定時刻に発火（朝/夜） | Heartbeat fixed-time（**実装済み**） |
| **間隔ベース** | 定期的にチェック | Heartbeat interval（**実装済み**） |
| **イベント接近** | カレンダー予定の N 分前 | Heartbeat + カレンダー照合（**未実装**） |
| **変化検出** | Web ページ/RSS の更新 | feed-check, web-monitor-check（**実装済み**） |
| **会話コンテキスト** | チャットの話題に連動 | メモリ自動注入（**部分実装**） |
| **閾値到達** | 未読/未消化が一定数を超えた | ストア件数チェック（**未実装**） |
| **期日接近** | goal の期日が迫っている | メモリの日付パース + 評価（**実装済み: deadlineParser + formatGoalsWithDeadlines**） |
| **パターン認識** | 行動パターンからの予測 | reflection 分析結果の活用（**未実装**） |
| **無活動検出** | 目標に対して進捗がない | `updatedAt` + goal の組み合わせ（**実装済み: F11/F12**） |

### 4.3 配信方法の要件

| 緊急度 | 配信方法 | ユースケース例 |
|--------|---------|-------------|
| **即時** | Push 通知 + HeartbeatPanel | 競合サイト変更検知（A2） |
| **次回閲覧時** | HeartbeatPanel（ピン留め） | 朝のブリーフィング（A1, B1, C1） |
| **会話内** | チャットレスポンスに付加 | コンテキスト型の関連情報（A4, C5） |
| **オンデマンド** | ユーザーが明示的に要求 | 進捗確認（B5, C2） |

### 4.4 頻度制御の要件

ペルソナごとの許容頻度の違いから、制御パラメータを導出する。

| パラメータ | ペルソナ A | ペルソナ B | ペルソナ C |
|-----------|----------|----------|----------|
| Push 通知/日 | 3-5 件（重要なもの） | 5-8 件（リマインド中心） | 1-2 件（ナッジ） |
| パネル内提案/日 | 10-15 件 | 8-10 件 | 3-5 件 |
| チャット内サジェスト | 高頻度（毎回） | 中頻度（関連時のみ） | 低頻度（学習時のみ） |
| 週次サマリー | 必要（情報整理） | 必要（進捗管理） | 強く必要（振り返り） |

**導出される制御機構:**
- `maxPushPerDay`: 日次 Push 通知の上限（デフォルト 5）
- `suggestionFrequency`: `high` | `medium` | `low`（チャット内サジェスト頻度）
- `digestSchedule`: `daily` | `weekly`（ダイジェスト配信頻度）
- `quietHours` の拡張: 曜日ベース（休日設定）

### 4.5 フィードバックの粒度

| レベル | UI | 用途 |
|--------|-----|------|
| **二値** | Accept / Dismiss | 全ての提案に対する基本フィードバック |
| **延期** | Snooze（1h / 明日 / 来週） | リマインド型・ナッジ型の時間調整 |
| **理由付き** | 「興味なし」「タイミングが悪い」「既に知っている」 | 提案品質の改善シグナル |
| **暗黙的** | 提案を開いた/開かなかった、滞在時間 | 低コストで大量のシグナル収集 |

## 7. PoC 優先順位と実装ロードマップ

### 優先度の判断基準

1. **自律型エージェントの価値を最も端的に実証できるか** — PoC の目的に直結する機能を最優先
2. **既存基盤の活用度** — 低コストで高インパクトを実現できるものから着手
3. **他の機能の前提条件になるか** — 後続機能のブロッカーを先に解消

### P0: これがないと PoC にならない

| 順位 | 機能 | 根拠 | 規模 | 既存基盤 |
|------|------|------|------|---------|
| **1** | **F9: ブリーフィング高度化** | 全ペルソナが朝のブリーフィングを起点にしている。既存の briefing-morning の**プロンプト改修だけ**で goal/context メモリ参照を追加でき、最速で「自律的に価値を届ける」体験を実現 | **小** | briefing-morning タスク |
| **2** | **F5+ MVP: 多段階 RSS フィルタリング** | ペルソナ A の核心。「200 件 → 必読 5 件」の圧縮は **PoC のヘッドライン**。MVP は feed-check の結果をメモリ（preference/goal）と照合して LLM が 3 段階（必読/推奨/スキップ）に分類するところまで | **中** | feed-check + memory |
| **3** | **F8: 会話内コンテキストサジェスト** | 全ペルソナで最高頻度（11 件/日）・最高 Accept 率。「ORM について聞いたら、クリップ + RSS + メモリを横断して回答する」体験は、既存ツールとの**最大の差別化**。instructionBuilder のメモリ注入を拡張し、clip/feed も検索対象に加える | **中** | getRelevantMemories + instructionBuilder |

**この 3 機能で実現できる PoC ストーリー**:

> 朝起きると、200 件の RSS が 5 件に圧縮されたブリーフィングが届く（F5+ + F9）。
> 出社して会議中に「Drizzle について最近何がある？」と聞くと、
> RSS 18 件 + クリップ 6 件 + メモリ 3 件を横断した統合回答が返ってくる（F8）。

## 8. 設計上の原則

ユースケース分析から導出した、提案エンジン全体を通じた設計原則。

### 8.1 「邪魔にならない」が最優先

CHI 2025 の知見（頻度↑ → 好感度半減）+ 3 ペルソナの「邪魔な場面」分析から:

- **デフォルトは控えめ**。ユーザーが「もっと欲しい」と思う頻度から始める
- **提案の抑制は提案の追加より重要**。フォーカスモード、quiet hours、日次上限を初期実装に含める
- **暗黙的フィードバック（開封/未開封）** を活用し、明示的フィードバックの負担を最小化

### 8.2 「忘れない」が差別化要因

3 ペルソナ共通のペインポイントは **「覚えていてくれない」** こと:

- 高橋: 「前に読んだあの記事が見つからない」
- 山田: 「帰宅後に "あれ忘れてた" が頻発」
- 佐藤: 「先月学んだことを振り返る習慣がない」

→ メモリの自動蓄積 + 適切なタイミングでの引き出しが核心体験。提案エンジンは **メモリの「出口」** として機能する。

### 8.3 「成長する」が長期リテンション

- 1 日目の提案と 30 日目の提案が同じでは価値がない
- reflection + フィードバックで提案が進化する体験が、継続利用の鍵
- ペルソナ C のシナリオが最も分かりやすい例（学習の進捗に応じた提案の変化）

### A-4. 3 ペルソナ横断の比較分析

#### エージェントの介入パターン

| 観点 | 高橋（テックリード） | 山田（PM） | 佐藤（学習者） |
|------|---------------------|-----------|---------------|
| **提案総数/日** | 16 件 | 11 件 | 10 件 |
| **Push 通知/日** | 2 件 | 2 件 | 1 件 |
| **処理する情報量** | RSS 200 件/日 → 約 15 件に圧縮 | カレンダー + メモリ中心 | RSS 少量 + クリップ + メモリ |
| **最も価値の高い提案類型** | コンテキスト型 `[X]` + ダイジェスト型 `[D]` | リマインド型 `[R]` | ナッジ型 `[N]` |
| **沈黙すべき時間帯** | 集中コーディング中 | MTG 中、週末 | 業務時間、リラックス中 |
| **エージェントの人格** | **200 件全部読んだ情報番人** | 有能な秘書 | 寄り添うコーチ |
| **情報の流れ** | 外部 200 件 → 多段階フィルタ → ユーザー | 過去 → 現在（リマインド） | 過去 → 未来（成長） |

**最重要の発見**:

1. **コンテキスト型 `[X]` が全ペルソナ合計で最多（11 件）かつ Accept 率最高**。会話の文脈に応じて過去の情報を引き出す F8 の実装が核心価値。

2. **ダイジェスト型 `[D]` がペルソナ A の改訂で重要度上昇（9 件）**。1 日 200 件の情報を圧縮する「多段階フィルタリング + トピック統合」は、単なる briefing 拡張ではなく、**独立した情報処理パイプライン** として設計すべき。→ **F5 を F5+（多段階 RSS フィルタリング + トピック統合）に拡張し、Tier 1 に格上げ**。

3. **「情報番人」モデル**: ペルソナ A のユースケースが示す核心は、エージェントが **200 件全部を読んだかのように振る舞い** 、ユーザーの問いに応じて任意の粒度で情報を引き出せること。これは「フィルタリング」の先にある「**情報のインデックス化と即座の検索**」機能を要求する。

#### 新規機能要件: F17 ソース横断トピック統合

ペルソナ A の 200 件/日シナリオから導出された、F5 に含まれない独立した機能要件:

| 要件 | 説明 |
|------|------|
| **トピック検出** | 異なるソース（RSS, HN クリップ, Web 監視）に分散した同一トピックを自動検出 |
| **重複排除** | 同一トピックを 1 エントリに統合（最も詳しい記事を主軸、他は差分抽出） |
| **ソースカウント** | 「4 ソースで言及」のように話題性の指標を付与 |
| **既読追跡** | RSS で既に見た記事を HN 経由で再提案しない |
| **クロスリファレンス** | 会話中に「Bun 1.2 について」と聞かれたら、RSS + クリップ + HN のすべてから統合回答 |
