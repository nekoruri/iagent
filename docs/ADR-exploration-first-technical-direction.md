# ADR: 探索優先の技術判断とレビュー方針

## 1. 背景

iAgent は、実装の完成度だけでなく、将来のエージェント像を探る研究・探索の段階にある。
この段階では、新しい LLM、planner、tool runtime、document / skill 形式が継続的に現れる。

そのため、短期 PoC の「検証成立」を最優先にすると、次の問題が起きやすい。

- まだ固定すべきでない schema や責務分割を早く固定してしまう
- 現行実装との整合を優先しすぎて、将来の差し替え余地を狭める
- 可能性の比較より、説明可能性や再現性の先回りに寄りすぎる

一方で、探索段階であっても、セキュリティ、権限境界、データ破壊、外部公開面の問題は引き続き重要である。

この ADR は、今後の提案・レビュー・設計判断における優先順位を定める。

---

## 2. 決定

今後の長期提案・初期設計・研究トラックの判断では、**探索優先**を基本姿勢とする。

優先順位は次の通り。

1. 将来の可能性を広げること
2. 学びを仕様・文書・契約・スキルとして資産化できること
3. LLM / planner / executor / tool runtime / storage を差し替えやすいこと
4. 後から安全性・運用性・検証性へ収束できること

短期 PoC の成立性、再現性、運用監査性は重要だが、探索段階では常に最優先には置かない。

---

## 3. レビューの基本モード

### 3.1 探索モード

提案書、長期構想、初期アーキテクチャ、研究トラック整理では、原則として探索モードで読む。

探索モードでは、次を優先して評価する。

- その設計が将来の選択肢を増やすか
- 失敗しても学びが他方式へ持ち越せるか
- コードより先に、仕様・契約・スキル・文書として知識を分離できるか
- 特定ベンダーや現行実装に閉じずに伸ばせるか

探索モードでは、次を初手から強く要求しすぎない。

- 現行アーキテクチャへの即時整合
- 早すぎる schema 固定
- 完全な testability / auditability
- PoC を短期完了させるための閉じた最適化

### 3.2 収束モード

次に当てはまる変更や文書では、収束モードを強める。

- 本番運用に近い実装
- 権限、秘密情報、外部公開面に触れる変更
- データ移行や破壊的変更を伴う設計
- source of truth になる schema / policy / contract の固定
- UI / storage / executor をまたぐ不可逆な結合

収束モードでは、次を優先して確認する。

- 安全境界
- permission / confirmation policy
- testability
- auditability
- operability
- backward compatibility

1 つの提案や変更の中で、探索モードと収束モードが混在してよい。
ただし、どこを探索として扱い、どこを固定判断として扱うかを明示する。

---

## 4. 堅さの重み付け

堅さは一律のブレーキとして扱わず、**重大度** と **可逆性** に応じて重み付けする。

### 4.1 強く止めるべき事項

次は探索段階でも防御的に指摘し、必要なら進行を止める。

- セキュリティ境界の後退
- 権限拡大を伴うのに policy が固定されていない設計
- ユーザーデータ、秘密情報、外部公開面の取り扱い不備
- データ消失や破壊的副作用のリスク
- 後戻りしにくい vendor lock-in
- 将来の runtime / model / tool 差し替えを著しく難しくする中核結合

### 4.2 注意しつつ進めてよい事項

次は将来の締めどころとして指摘するが、探索の進行自体は止めない。

- 一時的に粗い schema
- 未整備な observability / metrics / test coverage
- 暫定的な storage 形式や実行パイプライン
- 比較のために残す重複実装

### 4.3 探索上許容する事項

次は意図が明確なら許容する。

- まだ決め切らない語彙や責務境界
- 複数方式を並行比較するための冗長さ
- 収束前提ではない experiment-first な document / code path

---

## 5. 拡張性に対する防御的レビュー

探索優先であっても、拡張性を削る設計には防御的に入る。

特に次は早い段階で指摘する。

- 特定ベンダーや特定モデルの前提が中核 contract に漏れている
- 仕様、実行系、UI、保存形式が強く結合していて差し替えにくい
- 早期 migration 前提の schema 固定が将来の探索を縛る
- permission policy や tool contract が feature 個別実装に埋まる
- capability 追加時に再利用ではなくコピペ再実装が増える
- 将来の skill registry / planner / compiler / executor の差し替え余地が消える

拡張性の観点では、**今は動くが将来の比較可能性を潰す設計**をリスクとして扱う。

---

## 6. 実務上の運用ルール

### 6.1 提案を書くとき

- 「何を固定するか」より先に「どの可能性を開くか」を書く
- 未確定事項は欠陥として隠さず、探索余地として明示する
- 可逆な実験と不可逆な決定を分けて書く
- 学びを回収する単位を code ではなく spec / contract / skill / document として意識する

### 6.2 レビューするとき

- まず探索モードか収束モードかを明示する
- 指摘は「今止めるべき」「注意しつつ進めてよい」「探索上許容」に分ける
- PoC の成立性だけで設計の価値を狭く判定しない
- 一方で、安全性や拡張性を削る不可逆な判断には強く入る

### 6.3 実装へ落とすとき

- 探索段階では adapter / contract / boundary を厚くし、内部実装は差し替え可能に保つ
- 収束段階で schema、permission、audit、test を順に固定する
- 「今の便宜」で入れた結合が source of truth へ昇格しないように注意する

---

## 7. この ADR が置き換える前提

本 ADR 以降、長期提案や初期設計を読む際に、
**「PoC としてどれだけ厳密に検証できるか」を最初の評価軸に置くことをやめる。**

代わりに、最初の評価軸を次へ置き換える。

- 可能性を広げるか
- 学びを次へ持ち越せるか
- 将来の差し替えや横展開に耐えるか

ただし、セキュリティ、権限境界、破壊性、外部公開面、不可逆なロックインは例外として、引き続き強く扱う。

---

## 8. 結果

この ADR により、iAgent の提案・レビュー・設計判断は次の姿勢で統一する。

- 長期方向では探索を優先する
- 堅さは重大度と可逆性で重み付けする
- 拡張性を削る設計には防御的に指摘する
- 安全性と運用性は、必要な地点で明示的に収束させる

これを、今後の proposal、track 文書、設計レビュー、実装判断の共通前提とする。
