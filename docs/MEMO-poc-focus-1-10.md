## 1. North Star / KPI 設計

- 目的: 開発判断を「実装量」ではなく「価値提供」で揃える。
- PoCで重要な理由: PoC は本番品質よりも学習速度が重要。評価軸がないと、何が当たり機能か判断できない。
- 最小実装:
  - 主要 KPI を 3 つに絞る（例: 提案 Accept 率、7日継続率、通知経由の再訪率）。
  - 指標定義を 1 ページ化してチームで固定。
- 成功判定: 週次レビューで「KPI の上下に基づく意思決定」が実際に行われる。

## 2. ユーザー検証ループ

- 目的: ペルソナ仮説と実ユーザー行動のズレを早期に潰す。
- PoCで重要な理由: 体験価値はログだけでは分からない。特に「邪魔/助かる」の感情は定性確認が必要。
- 最小実装:
  - 3 ペルソナ相当のユーザーに週次ヒアリング。
  - 「よかった提案 / 不要だった提案 / 欲しい提案」をテンプレで収集。
- 成功判定: 毎週 1 つ以上、ユーザー観測由来の改善が実装される。

## 3. オンボーディング最適化

- 目的: 初回セットアップの離脱を減らし、価値体験までの時間を短縮する。
- PoCで重要な理由: PoC は継続利用データが生命線。初回でつまずくと検証母数が減る。
- 最小実装:
  - ペルソナ別プリセット（情報収集型 / PM型 / 学習者型）を用意。
  - 初回は推奨プリセット 1 クリック適用。
- 成功判定: 初回 24 時間以内のアクティブ率が改善する。

## 4. 信頼性 SLO 運用

- 目的: 「動くはず」ではなく「どの程度安定して動いているか」を見える化する。
- PoCで重要な理由: 自律エージェントは、失敗すると体験価値が急落する。信頼性の最低ラインが必要。
- 最小実装:
  - SLO を 3 指標で定義（Heartbeat 成功率、Push 到達率、提案生成遅延）。
  - しきい値を切ったときの暫定アクションを決める。
- 成功判定: 失敗傾向を週内に検知し、改善チケット化できる。

## 5. 通知パーミッション回復導線

- 目的: OS/ブラウザ側の通知権限変化で機能停止する問題を減らす。
- PoCで重要な理由: 通知はプロアクティブ体験の入口。許可状態が壊れると価値検証が止まる。
- 最小実装:
  - 権限状態の定期チェック。
  - `denied` / `default` / `granted` 別に UI 導線を出し分ける。
- 成功判定: 「通知が来ない」起因の離脱報告が減る。

## 6. コスト制御レイヤー

- 目的: 推論コストの暴走を抑えつつ、体験品質を維持する。
- PoCで重要な理由: PoC では試行回数が増えるため、1回あたりのコスト最適化が継続速度を左右する。
- 最小実装:
  - タスクごとにモデルグレードとトークン予算を定義。
  - 予算超過時の縮退動作（要約短縮、次回回し）を決める。
- 成功判定: 週次のコスト予実差が小さくなり、異常増加が事前検知される。
- 実装メモ（2026-03-05）:
  - `heartbeatCost.ts` でタスク別実行ポリシー（`gpt-5-nano` / `gpt-5-mini` + 出力トークン上限）を定義。
  - Layer1/2/3 で共通に日次トークン予算を判定し、逼迫時は縮退モード・非クリティカル次回回しを適用。
  - `ops-events` の `heartbeat-run` に token 使用量を記録し、当日使用量を再利用して次回判定に反映。

## 7. 記憶品質ガバナンス

- 目的: 記憶の精度と信頼性を保つ（誤記憶・古い記憶の悪影響を減らす）。
- PoCで重要な理由: プロアクティブ提案は記憶品質に直結。低品質記憶は提案品質を連鎖的に下げる。
- 最小実装:
  - 「記憶の修正」「重要度調整」「無効化」をユーザー操作可能にする。
  - 一定期間未参照・低評価の記憶を再評価対象にする。
- 成功判定: 記憶起因の誤提案報告が減り、提案 Accept 率が安定する。
- 実装メモ（2026-03-05）:
  - `memoryStore` に `updateMemory` / `archiveMemory` / `listMemoryReevaluationCandidates` を追加。
  - `MemoryPanel` に編集フォーム（内容/重要度/タグ）と手動無効化ボタンを追加。
  - 低重要度かつ長期間未参照（既定 14 日）の記憶を「再評価候補」として表示。
  - `memory` ツールに `update` / `archive` / `reevaluate` を追加し、会話経由でも記憶ガバナンス操作を実行可能化。
  - Heartbeat Worker ツールに `listMemoryReevaluationCandidates` を追加し、reflection タスクから見直し候補を参照可能化。

## 8. データポータビリティ

- 目的: 端末変更やブラウザ消去時にデータ喪失を防ぐ。
- PoCで重要な理由: 検証期間中に履歴が消えると、長期学習の評価が不可能になる。
- 最小実装:
  - 設定・メモリ・会話のエクスポート/インポートを提供。
  - 復元手順を1画面で完了できるようにする。
- 成功判定: データ移行時の再セットアップ時間が大幅に短縮される。
- 実装メモ（2026-03-05）:
  - `core/dataPortability.ts` を追加し、`config / conversation-meta / conversations / memories / memories_archive / attachments` を単一 JSON 形式で export/import 可能化。
  - import 時はフォーマット検証（`schemaVersion=1`）後に対象ストアを置換復元し、設定は localStorage + IndexedDB の両方へ同期。
  - `SettingsModal` の「ストレージ」セクションに `データをエクスポート` / `データをインポート` / `再読み込みして反映` を追加。
  - 復元対象件数（会話・メッセージ・記憶・アーカイブ・添付）を UI で表示し、実行結果の可視性を確保。

## 9. 拡張エコシステム戦略

- 目的: 「つながる先」を増やし、ユースケースの幅を広げる。
- PoCで重要な理由: 単体機能の完成度より、外部連携の再現性が PoC の説得力を上げる。
- 最小実装:
  - MCP プリセットをカテゴリ別に整備。
  - 最低限の推奨セット（例: GitHub/Notion/RSS）を即時導入可能にする。
- 成功判定: 外部連携を使った利用シナリオの再現率が上がる。
- 実装メモ（2026-03-05）:
  - `SettingsModal` の MCP セクションに「クイック追加（MCPプリセット）」を追加。
  - プリセットをカテゴリ表示付きで実装（GitHub / Notion / RSS Reader / Slack / Gmail / Google Calendar）。
  - `推奨セットを追加` で GitHub・Notion・RSS Reader を一括追加可能化。
  - 重複追加は名前/URL ベースで自動スキップし、追加結果（成功/スキップ）を UI に表示。

## 10. セキュリティの次段階

- 目的: PoC のスピードを維持しつつ、事故確率を抑える。
- PoCで重要な理由: デモ段階でもキー漏えい・誤実行は信頼を損なう。最低限のガードは先に必要。
- 最小実装:
  - API キー管理の見直し（保存/表示/更新時の安全策）。
  - 自動実行ログ（いつ、何が、なぜ動いたか）を記録・閲覧可能にする。
  - 権限の最小化（必要な機能のみ有効化）を徹底。
- 成功判定: 監査ログで主要自動実行を追跡でき、セキュリティ起因の重大インシデントが発生しない。
- 実装メモ（2026-03-05）:
  - `SettingsModal` の Heartbeat セクションに「自動実行ログ（Action Planning）」を追加し、最新 20 件を UI 表示。
  - ログ項目ごとに「アクション種別 / 実行時刻 / reason / detail」を表示し、`再読み込み` ボタンで即時更新可能化。
  - アクション種別は `toggle-task` 等の内部名を日本語ラベル（タスク切替・間隔変更・静寂時間・静寂曜日）へ変換。
  - API キー欄は保存済み値を直接再表示せず、空欄入力 + 状態表示（保存済み/更新予定/削除予定）で運用可能に変更。
  - 保存済みキーごとに `保存済みキーを削除` / `削除を取り消す` を追加し、保存時に保持・更新・削除を明示的に制御。
  - 基本設定に `最小権限プリセットを適用` を追加し、Heartbeat 有効/通知・音声入出力・プロキシ・MCP 接続を一括で無効化できるようにした。

## 推奨実行順（PoC）

1. まず 1, 2, 4 を先行（評価軸 + ユーザー学習 + 信頼性監視）
2. 次に 3, 5, 6（体験入口 + 通知復旧 + コスト最適化）
3. 最後に 7, 8, 9, 10（品質維持と拡張性）

この順番で進めると、短期で「学習速度」と「体験の再現性」を同時に高めやすい。
